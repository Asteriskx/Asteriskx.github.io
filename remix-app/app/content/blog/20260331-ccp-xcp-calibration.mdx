<h2 id="s1">01. CCP / XCP とは</h2>

CCP（CAN Calibration Protocol）と XCP（Universal Measurement and Calibration Protocol）は、**ECU 内部変数のリアルタイム計測（DAQ）とキャリブレーション（パラメータ書き換え）**を行うためのプロトコル。
ASAM（Association for Standardization of Automation and Measuring Systems）が規格化している。

| プロトコル | 規格 | 位置づけ |
|-----------|------|---------|
| **CCP** | ASAM MCD-1 CCP | CAN 専用の旧世代プロトコル |
| **XCP** | ASAM MCD-1 XCP | CAN / CAN FD / Ethernet / USB など複数トランスポートに対応した後継 |

UDS が「診断・書き込み」用途であるのに対し、XCP は**開発・適合段階でのリアルタイム観測と値の書き換え**に特化している。
ツールとしては Vector の CANape や ETAS の INCA が代表的。

---

<h2 id="s2">02. プロトコルスタック上の位置</h2>

```
┌──────────────────────────────────────────┐
│  XCP（ASAM MCD-1 XCP）                   │  ← アプリケーション層
│  計測・キャリブレーション定義             │
├──────────────────────────────────────────┤
│  トランスポート層                         │
│  XCP on CAN / XCP on CAN FD              │
│  XCP on Ethernet（TCP or UDP）           │
│  XCP on USB / SPI など                   │
├──────────────────────────────────────────┤
│  CAN / CAN FD / Ethernet など            │  ← データリンク + 物理層
└──────────────────────────────────────────┘
```

XCP はトランスポート層を抽象化しているため、**上位のコマンド体系はどのトランスポートでも共通**。
CCP は CAN 専用であり XCP の前身にあたる。現在の新規開発では XCP on CAN が主流。

### 車両ネットワーク上の XCP 全体像

```
  ┌─────────────────────────────────────┐
  │  適合 PC                            │
  │  CANape / INCA                      │
  └────────────────┬────────────────────┘
                   │ USB / Ethernet
                   │
  ┌────────────────┴────────────────────┐
  │  CAN インターフェース                │
  │  （Vector VN シリーズ など）         │
  └────────────────┬────────────────────┘
                   │
  ════════════════ CAN バス ════════════════════════
         │                   │                   │
  ┌──────┴───────┐   ┌───────┴──────┐   ┌────────┴─────┐
  │  Engine ECU  │   │  Brake ECU   │   │   Body ECU   │
  │              │   │              │   │              │
  │  XCP スレーブ│    │  XCP スレーブ│   │  （XCP なし） │
  │  ● DAQ       │   │  ● DAQ       │   │              │
  │  ● Calib     │   │  ● Calib     │   │              │
  └──────────────┘   └──────────────┘   └──────────────┘
```

適合ツールから見ると「CAN ID で区別された複数の XCP スレーブ」が並ぶ構成。
`CONNECT` コマンドのステーションアドレスで接続先 ECU を切り替える。

<h2 id="s3">03. CCP の概要</h2>

CCP は 2000 年代前後の ECU 開発で広く使われた CAN 専用プロトコル。
現在は XCP に置き換わっているが、既存 ECU との通信でまだ遭遇する。

### 通信モデル

CCP はマスター（PC・ツール側）とスレーブ（ECU 側）の 1:1 通信。

```
マスター（CANape / INCA）           スレーブ（ECU）
   │                                    │
   │── CTO（Command Transfer Object）→  │  コマンド送信
   │                                    │  処理
   │←─ CRM（Command Return Message）──   │  応答
   │                                     │
   │←─ DTO（Data Transfer Object）────   │  DAQ データ送信（非同期）
```

### CAN ID の割り当て

CCP では **コマンド用とレスポンス用に別々の CAN ID** を使用する。
接続時に `CONNECT` コマンドでスレーブアドレスを指定し、セッションを確立する。

| オブジェクト | 方向 | 内容 |
|------------|------|------|
| CTO | マスター → スレーブ | コマンド（最大 8 バイト）|
| CRM | スレーブ → マスター | コマンドへの応答（最大 8 バイト）|
| DTO | スレーブ → マスター | DAQ データ（非同期・最大 8 バイト）|

### CCP のパケット構造

```
Byte 0      Byte 1      Byte 2-7
[CMD/RET]   [CTR]       [データ]
  │           │
  │           └─ カウンタ（リクエスト / レスポンス対応確認用）
  └─────────── コマンドコード または 戻り値
```

---

<h2 id="s4">04. XCP の概要・パケット構造</h2>

XCP は CCP の後継として ASAM が策定。
CAN 以外のトランスポートにも対応し、機能も大幅に拡張されている。

### XCP の特徴

| 項目 | CCP | XCP |
|------|-----|-----|
| トランスポート | CAN のみ | CAN / CAN FD / Ethernet / USB / SPI など |
| ペイロード上限 | 8 バイト（CAN 制約） | トランスポート依存（CAN FD なら最大 64 バイト）|
| セキュリティ | Seed & Key | Seed & Key（アルゴリズム拡張）|
| STIM（逆方向注入） | 限定的 | 標準サポート |
| タイムスタンプ | なし | DAQ パケットに付加可能 |

### XCP パケット構造（XCP on CAN の場合）

```
Byte 0      Byte 1      Byte 2 ～
[PID]       [ERR/CTR]   [データ]
  │
  └─ Packet Identifier
       0x00-0xBF : コマンド / レスポンス
       0xC0-0xFF : DAQ パケット（DAQ リスト番号）
```

### レスポンスの種類

| PID | 種別 | 説明 |
|-----|------|------|
| 0xFF | **POSITIVE RESPONSE（RES）** | コマンド成功 |
| 0xFE | **ERROR（ERR）** | コマンド失敗。Byte 1 にエラーコード |
| 0xFD | **EVENT（EV）** | ECU からの非同期イベント通知 |
| 0xFC | **SERVICE REQUEST（SERV）** | ECU からのサービス要求 |

<h2 id="s5">05. DAQ（データ計測）</h2>

DAQ（Data Acquisition）は XCP の中核機能。
ECU 内部の変数（センサー値・制御量・フラグなど）を**指定した周期でリアルタイムに読み出す**仕組み。

### ツール・XCP スレーブ・ECU 内部タスクの 3 層構造

```
  ┌─────────────────────┐
  │  CANape / INCA      │  ← 適合ツール（マスター）
  │                     │     A2L で変数名 ↔ アドレスを把握
  └──────────┬──────────┘
             │ XCP on CAN（コマンド / DAQ パケット）
             │
  ┌──────────┴──────────────────────────────────────────┐
  │  XCP スレーブ（ECU 内 XCP ドライバ）                  │
  │                                                     │
  │  CONNECT / GET_SEED / UNLOCK ─→ セッション確立       │
  │  SET_DAQ_PTR / WRITE_DAQ ──────→ DAQ 設定受け付け    │
  │  START_STOP_SYNCH ─────────────→ 計測開始            │
  │  SHORT_DOWNLOAD ───────────────→ パラメータ書換       │
  │                                                      │
  │  ┌──────────────────────────────────────────────┐    │
  │  │  制御タスク（例: 1ms 周期）                    │   │
  │  │                                              │   │
  │  │  ① センサー読み込み                           │   │
  │  │  ② 制御演算                                   │   │
  │  │     └→ XCP が変数値をスナップショット ───────────→ DAQ パケット送信
  │  │  ③ アクチュエータ出力                          │   │
  │  │     ↑ SHORT_DOWNLOAD で書き換えた値が反映      │   │
  │  └──────────────────────────────────────────────┘   │
  └─────────────────────────────────────────────────────┘
```

XCP スレーブは制御タスクと**非同期に**動作する。
制御タスクが 1ms で回っていても、XCP のコマンド処理は別タスク（低優先度）で動くのが一般的。
DAQ のスナップショットは制御タスクの中でトリガされるため、制御周期に同期した計測が実現できる。

### DAQ の動作イメージ

```
マスター                              スレーブ（ECU）
   │                                      │
   │── DAQ リスト設定（変数・周期）──→      │
   │── START_STOP_SYNCH ─────────────→    │
   │                                      │
   │←─ DAQ パケット（変数値）──────────     │  1ms ごと
   │←─ DAQ パケット（変数値）──────────     │  1ms ごと
   │         ：                            │
```

通常の `UPLOAD` コマンドによる読み出しはポーリング（マスターがリクエストして ECU が応答）だが、
DAQ は ECU が**自律的にデータを送信**する点が大きく異なる。
制御周期に同期した高頻度計測が可能（CAN FD では数百 Hz も現実的）。

### ポーリングと DAQ の比較

| 方式 | 仕組み | 適用場面 |
|------|--------|---------|
| ポーリング（UPLOAD）| マスターが都度リクエスト | 単発確認・低頻度読み出し |
| DAQ | ECU が自律送信 | 高頻度・周期的な計測 |

<h2 id="s6">06. ODT と DAQ リスト</h2>

DAQ の設定は **ODT（Object Descriptor Table）** と **DAQ リスト** の 2 階層で構成される。

### 階層構造

```
DAQ リスト 0（例: 1ms 周期）
  └─ ODT 0
       ├─ Entry 0: 変数 A のアドレス・サイズ
       ├─ Entry 1: 変数 B のアドレス・サイズ
       └─ Entry 2: 変数 C のアドレス・サイズ
  └─ ODT 1
       ├─ Entry 0: 変数 D のアドレス・サイズ
       └─ Entry 1: 変数 E のアドレス・サイズ

DAQ リスト 1（例: 10ms 周期）
  └─ ODT 0
       └─ Entry 0: 変数 F のアドレス・サイズ
```

| 用語 | 説明 |
|------|------|
| **DAQ リスト** | 計測グループ。送信周期・イベントを設定する単位 |
| **ODT** | CAN フレーム 1 本に対応するデータのまとまり（最大 7 バイト / CAN フレーム）|
| **ODT Entry** | 計測する変数 1 つ。ECU メモリのアドレスとサイズで指定 |

### DAQ 設定のシーケンス

```
FREE_DAQ                        ← 既存 DAQ リストを全解放
ALLOC_DAQ      daqCount         ← DAQ リスト数を確保
ALLOC_ODT      daqNum, odtCount ← ODT 数を確保
ALLOC_ODT_ENTRY daqNum, odtNum, entryCount ← ODT エントリ数を確保

SET_DAQ_PTR    daqNum, odtNum, entryNum     ← 書き込み先を指定
WRITE_DAQ      bitOffset, size, address     ← 変数アドレスを登録
  （WRITE_DAQ を変数の数だけ繰り返す）

SET_DAQ_LIST_MODE  daqNum, eventChannel, prescaler ← 周期を設定
START_STOP_DAQ_LIST  daqNum, mode=START            ← 個別スタート
START_STOP_SYNCH     mode=START_SELECTED           ← 一斉スタート
```

<h2 id="s7">07. キャリブレーション</h2>

キャリブレーションは ECU 内部の**パラメータ（マップ・定数）をリアルタイムに書き換える**機能。
制御ゲイン・センサー補正値・マップテーブルなどを走行しながら調整できる。

### MTA（Memory Transfer Address）

XCP のメモリ操作は **MTA（Memory Transfer Address）** というポインタを経由して行う。
`SET_MTA` で書き込み先アドレスを設定してから `DOWNLOAD` / `UPLOAD` を実行する。

```
マスター                        スレーブ
   │── SET_MTA（アドレス A） →  │  ポインタを A にセット
   │── DOWNLOAD（値） ────────→ │  A に値を書き込み
   │   または
   │── UPLOAD（サイズ） ──────→ │
   │←─ RES（読み出しデータ） ──  │  A からサイズバイト読み出し
```

### SHORT_UPLOAD / SHORT_DOWNLOAD

4 バイト以下の小さなデータはアドレスをパケット内に直接埋め込む `SHORT_UPLOAD` / `SHORT_DOWNLOAD` が使える。
`SET_MTA` を省略できるため、スポット的な確認・修正に便利。

### RAM キャリブレーションとフラッシュ書き込みの違い

| 方式 | 説明 | 特徴 |
|------|------|------|
| **RAM キャリブレーション** | RAM 上のパラメータを直接書き換え | 即時反映・電源 OFF で消える |
| **RAM init / Flash** | RAM で調整後、確定値をフラッシュに焼く | 永続化。リプロと組み合わせる |

開発フェーズでは RAM キャリブレーションで素早く値を試し、確定後にフラッシュへ書き込む流れが一般的。

<h2 id="s8">08. A2L ファイル</h2>

A2L（ASAM MCD-2 MC）は、**ECU 内部の変数やパラメータのレイアウトを記述したデータベースファイル**。
拡張子は `.a2l`。CANape・INCA などのツールはこのファイルを読み込むことで ECU の変数名とメモリアドレスを紐付ける。

### A2L の主要オブジェクト

| オブジェクト | 説明 |
|------------|------|
| **MEASUREMENT** | 計測変数（読み出し専用）。センサー値・計算値など |
| **CHARACTERISTIC** | キャリブレーションパラメータ（読み書き可）。マップ・定数など |
| **AXIS_PTS** | マップの軸データ（ブレークポイント） |
| **COMPU_METHOD** | 生値 ↔ 物理値の変換式（係数・オフセット・単位） |
| **MEMORY_SEGMENT** | RAM / Flash などのメモリ領域定義 |

### A2L の記述例（簡略）

```
/begin MEASUREMENT
  engineRpm          /* 変数名 */
  "エンジン回転数"    /* 説明 */
  UWORD              /* データ型 */
  convRpm            /* COMPU_METHOD 参照 */
  0                  /* 下限 */
  8000               /* 上限 */
  ECU_ADDRESS 0x40001234  /* メモリアドレス */
/end MEASUREMENT

/begin COMPU_METHOD convRpm
  "rpm 換算" RAT_FUNC "%6.1" "rpm"
  COEFFS 0 0.125 0 0 0 1  /* 物理値 = 生値 × 0.125 */
/end COMPU_METHOD
```

### A2L の生成フロー

```
ECU ソースコード
  │
  ▼
コンパイル・リンク
  │
  ├─→ .elf / .hex（フラッシュ書き込み用）
  │
  └─→ .a2l（シンボル情報 + 手動記述のメタ情報）
        │
        └─→ CANape / INCA に読み込み → 計測・キャリブレーション開始
```

AUTOSAR 環境では コード生成ツールが A2L を自動生成するケースも多い。

<h2 id="s9">09. 主要コマンド一覧</h2>

### 接続・セッション管理

| コマンド | 説明 |
|---------|------|
| **CONNECT** | セッション確立。ECU の基本情報（バイトオーダー・アドレス粒度など）を取得 |
| **DISCONNECT** | セッション切断 |
| **GET_STATUS** | ECU の現在状態（セッション・DAQ 状態・キャリブレーションモードなど）取得 |
| **GET_COMM_MODE_INFO** | 通信モード情報（インタリーブモード・ブロックモード対応など）取得 |
| **GET_ID** | ECU 識別情報取得（A2L ファイル名など）|
| **TIME_CORRELATION** | マスター / スレーブ間のタイムスタンプ同期 |

### セキュリティ

| コマンド | 説明 |
|---------|------|
| **GET_SEED** | セキュリティ解除用シードを取得 |
| **UNLOCK** | キーを送信してセキュリティ解除 |

UDS の SecurityAccess（0x27）と同様の Seed & Key 方式。
キャリブレーション・フラッシュ書き込みなど保護リソースへのアクセス前に実施する。

### メモリアクセス

| コマンド | 説明 |
|---------|------|
| **SET_MTA** | MTA（メモリ転送ポインタ）を設定 |
| **UPLOAD** | MTA から指定バイト数を読み出し |
| **SHORT_UPLOAD** | アドレス直指定で 4 バイト以下を読み出し |
| **DOWNLOAD** | MTA へ指定バイト数を書き込み |
| **SHORT_DOWNLOAD** | アドレス直指定で 4 バイト以下を書き込み |
| **DOWNLOAD_NEXT** | ブロック転送モードで連続書き込み |
| **MODIFY_BITS** | ビットマスクで部分書き換え |

### DAQ 設定・制御

| コマンド | 説明 |
|---------|------|
| **FREE_DAQ** | 全 DAQ リストを解放 |
| **ALLOC_DAQ** | DAQ リストを確保 |
| **ALLOC_ODT** | ODT を確保 |
| **ALLOC_ODT_ENTRY** | ODT エントリを確保 |
| **SET_DAQ_PTR** | DAQ 設定の書き込み先を指定 |
| **WRITE_DAQ** | ODT エントリに変数を登録 |
| **SET_DAQ_LIST_MODE** | DAQ リストの周期・イベントを設定 |
| **START_STOP_DAQ_LIST** | DAQ リストの個別開始 / 停止 |
| **START_STOP_SYNCH** | 複数 DAQ リストの一斉開始 / 停止 |
| **GET_DAQ_CLOCK** | ECU のタイムスタンプカウンタを取得 |

### フラッシュ書き込み（PGM）

| コマンド | 説明 |
|---------|------|
| **PROGRAM_START** | プログラミングモード開始 |
| **PROGRAM_CLEAR** | フラッシュ消去 |
| **PROGRAM** | フラッシュ書き込み |
| **PROGRAM_RESET** | ECU リセット |

<h2 id="s10">10. 実務での使い方</h2>

### 開発フェーズ別の使い分け

| フェーズ | 主な用途 |
|----------|---------|
| **制御ロジック開発** | DAQ で内部変数を観測しながらロジックを検証 |
| **パラメータ適合** | CHARACTERISTIC を RAM キャリブレーションでリアルタイム調整 |
| **HIL / SIL テスト** | STIM で入力信号を注入し、ECU の応答を DAQ で確認 |
| **量産前確定** | 調整済みパラメータを PGM コマンドでフラッシュに焼き付け |

### CANape / INCA での典型的なワークフロー

```
1. A2L ファイルと .hex を CANape に読み込み
   └→ CONNECT → GET_ID で A2L ファイル名確認

2. 計測設定
   └→ 観測したい MEASUREMENT 変数を選択
   └→ FREE_DAQ → ALLOC_DAQ → ... → START_STOP_SYNCH

3. 走行・計測
   └→ DAQ パケットが自動的に届き波形表示

4. キャリブレーション
   └→ GET_SEED / UNLOCK でセキュリティ解除
   └→ CHARACTERISTIC のマップをグリッドで編集
   └→ SHORT_DOWNLOAD で即時書き換え・挙動確認

5. 確定値の保存
   └→ PROGRAM_START → PROGRAM_CLEAR → PROGRAM
   └→ PROGRAM_RESET で ECU リセット・動作確認
```

### よくあるトラブルと対処

- **CONNECT 後に ERR_ACCESS_LOCKED**
XCP セッションが前回の切断で正常に終了していない場合がある。
ECU を再起動して再接続するか、`DISCONNECT` を先に送って状態をリセットする

- **DAQ データが間引かれる / 欠落する**
CAN バスの負荷が高い場合に DAQ パケットがドロップする。
DAQ リストの変数数を減らすか、周期を遅くしてバス負荷を下げる。
CAN FD への切り替えも有効

- **SHORT_DOWNLOAD 後に値が戻る**
RAM キャリブレーション中に ECU の初期化処理がパラメータを上書きしている可能性。
書き換えが有効になるセクション（初期化後のタスク起動以降）のタイミングで実施する

- **A2L のアドレスがビルドごとにずれる**
最適化オプションやリンカスクリプトの変更でシンボルアドレスが変わる。
A2L は必ずその .elf と同一ビルドのものを使う

---

> **まとめ**
>
> - CCP は CAN 専用の旧世代プロトコル。XCP はマルチトランスポート対応の後継
> - XCP のコア機能は DAQ（周期的な変数読み出し）とキャリブレーション（パラメータ書き換え）
> - DAQ は ODT / DAQ リストの 2 階層で変数と送信周期を管理する
> - メモリアクセスは MTA ポインタを経由。SHORT 系コマンドで省略可能
> - A2L ファイルが変数名とメモリアドレスの橋渡しをしており、ツールと ECU を繋ぐ
> - Seed & Key でセキュリティ保護。キャリブレーション前に解除が必要
