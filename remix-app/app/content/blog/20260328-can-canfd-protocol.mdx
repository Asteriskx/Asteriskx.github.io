<h2 id="s1">01. 車載ネットワーク概要</h2>

### ECU とは

ECU（Electronic Control Unit）は車載コンピュータの総称。
センサーからの入力をもとに制御判断を行い、アクチュエータへ指令を出す。
現代の量産車には数十〜100 以上の ECU が搭載されており、これらが CAN バスを通じて互いに連携している。

用途に応じて専用の ECU（エンジン制御・ブレーキ制御・ボディ制御など）が設けられており、それぞれが担当領域のリアルタイム制御に特化した設計になっている。

### ネットワーク全体図

現代の自動車は用途ごとに **複数の CAN バスを分離し、Gateway ECU で橋渡しする** 構成が一般的。
バスを分けることで、ドメインをまたいだノイズ・障害伝播を防ぎ、セキュリティの境界も引きやすくなる。

```
                        ┌─────────────────────────────────────────────┐
                        │              Gateway ECU                    │
                        │  （バス間ルーティング・プロトコル変換）         │
                        └──────┬──────────┬──────────┬───────────────┘
                               │          │          │
              ┌────────────────┘          │          └──────────────────┐
              │                           │                             │
   ┌──────────▼──────────┐   ┌────────────▼────────────┐   ┌──────────▼──────────┐
   │  パワートレイン CAN  │   │      ボディ CAN          │   │    ADAS / Safety    │
   │   500 kbit/s        │   │   125〜250 kbit/s        │   │    CAN FD / Eth     │
   ├─────────────────────┤   ├─────────────────────────┤   ├─────────────────────┤
   │ ・エンジン ECU      │   │ ・ボディ制御 ECU         │   │ ・カメラ ECU        │
   │ ・トランスミッション │   │ ・ドア / ウィンドウ      │   │ ・レーダー ECU      │
   │ ・インバータ（EV）  │   │ ・ライト / ワイパー      │   │ ・LiDAR             │
   │ ・ABS / ESC         │   │ ・エアコン ECU           │   │ ・ADAS 統合 ECU     │
   │ ・EPS ECU           │   │                         │   │                     │
   └─────────────────────┘   └──────────┬──────────────┘   └─────────────────────┘
                                         │ LIN バス
                              ┌──────────▼──────────┐
                              │  LIN スレーブ群      │
                              │  （シート・ミラー等）  │
                              └─────────────────────┘
```

### 主要 ECU の役割

**エンジン ECU（ECM）**

パワートレインの中核。
燃料噴射量・点火タイミング・スロットル開度などを制御する。
車速・エンジン回転数・冷却水温といった基本情報を CAN バス上に周期的に配信しており、他の ECU がこれを参照して協調動作する。

**EPS ECU（電動パワーステアリング）**

車速・操舵角を受信し、アシスト量をリアルタイムに計算する。
ESC / VSA と連携して横滑り防止にも関与する。
ADAS 連携ではレーンキープ・自動操舵の操舵指令を CAN FD 経由で受け取るケースが増えており、Steer-by-Wire ではさらに厳密なリアルタイム性が求められる。

**トランスミッション ECU（TCU）**

AT / CVT / DCT などの変速制御を担う。
エンジン ECU とはトルク要求・シフトタイミングを CAN でやり取りし、変速ショックの低減やエンジンブレーキ協調を実現する。
EV / HEV ではモーターとエンジンの切り替え制御にも関わり、インバータ ECU との連携が加わる。

**ABS / ESC ECU**

ブレーキ制御と車両安定性の要。
各輪の車輪速センサーから回転数を取得し、ロック防止（ABS）・横滑り防止（ESC）・トラクションコントロールを行う。
EPS・ADAS と CAN で協調しており、自動緊急ブレーキ（AEB）では ADAS 統合 ECU からの減速指令を受けて制動を実行する。

**インバータ ECU（EV / HEV）**

EV・HEV のモーター駆動を制御する、エンジン ECU に代わるパワートレインの中核。
バッテリーの DC を AC に変換してモーターを回し、回生ブレーキ時は AC → DC 変換で回収する。
高電圧系（数百 V）を扱うため、CAN 通信で異常検出・緊急停止の応答性が重要になる。

**ボディ ECU（BCM）**

ドア・ライト・ワイパー・パワーウィンドウなどのボディ系デバイスを統合管理する。
速度よりコスト重視の領域で、ボディ CAN（125〜250 kbit/s）に接続されることが多い。
シート・ミラーなど低速で済むスレーブ群は LIN で束ね、BCM がマスターとしてゲートウェイ的に動作する構成が一般的。

**ADAS 統合 ECU**

カメラ・ミリ波レーダー・LiDAR から受け取ったデータをセンサーフュージョンし、衝突回避・レーンキープ・ACC などの判断を行う。
扱うデータ量が大きいため、CAN FD や Automotive Ethernet で接続されることが多い。
EPS や ESC への制御指令を出す側であり、車両安全に直結するため ASIL-D（ISO 26262 最高レベル）が求められることもある。

### 診断インターフェース（OBD-II）

OBD-II（On-Board Diagnostics II）は、1996 年に米国で乗用車への搭載が義務化された車両診断の標準規格。
欧州では EOBD として 2001 年から義務化されており、現在は世界共通の診断インターフェースとして定着している。

物理的には **16 ピンの DLC（Diagnostic Link Connector）** として車内（多くはハンドル下）に設置されており、診断ツールや PC を繋ぐ外部アクセス口になっている。
通信の下位レイヤーには CAN（ISO 15765-4）が使われており、診断ツールは CAN バス経由で各 ECU とやり取りする。

```
診断ツール（PC / スキャナ）
     │
  DLC コネクタ（OBD-II ポート）
     │
  Gateway ECU ─── 各ドメイン CAN バスへルーティング
```

OBD-II が定義するのはあくまで **物理ポートと基本診断データ（排ガス・MIL 等）のアクセス方法** まで。
ECU のフラッシュ書き込みや内部パラメータの読み書きには、上位プロトコルの **UDS（ISO 14229）** を使う。
実務では「OBD-II ポート経由で UDS サービスを発行する」という形が標準的。

厳密には OBD-II（ISO 15031 / SAE J1979）と UDS（ISO 14229）は別プロトコルだが、実務では OBD-II ポート経由で UDS サービスを使うのが一般的。

| サービス ID（SID） | 意味 |
|-------------------|------|
| 0x10 | DiagnosticSessionControl（診断セッション開始） |
| 0x11 | ECUReset |
| 0x22 | ReadDataByIdentifier（DID 読み出し） |
| 0x2E | WriteDataByIdentifier（DID 書き込み） |
| 0x27 | SecurityAccess（セキュリティ解除） |
| 0x34 / 0x36 / 0x37 | RequestDownload / TransferData / RequestTransferExit（フラッシュ書き込み） |
| 0x19 | ReadDTCInformation（DTC 読み出し） |
| 0x14 | ClearDiagnosticInformation（DTC 消去） |

ECU 開発中は **0x22 / 0x2E** で内部変数の読み書きデバッグ、リリース前は **0x34〜0x37** でフラッシュ書き込み確認、量産後は **0x19** で DTC 確認、という流れが基本になる。
UDS / DTC の詳細は[こちらの記事](/blog/20260328-uds-dtc-vehicle-diagnostics)にまとめている。

<h2 id="s2">02. CAN とは</h2>

CAN（Controller Area Network）は、1986 年に Bosch が開発したシリアル通信プロトコル。
もともとは自動車内の ECU 同士を繋ぐために設計された。
それまでは ECU ごとに専用ハーネスを引いていたが、CAN バス 1 本にすべてぶら下げる構成にすることで配線を大幅に削減できる。

現在は ISO 11898 として標準化されており、車載だけでなく産業機器・医療機器・ロボットなど幅広い分野で使われている。

### 車載ネットワークの変遷

1980 年代は各メーカーが独自プロトコルを使っていたが、CAN の登場で業界が一本化。
その後、帯域不足を補う形で CAN FD・Automotive Ethernet へと進化してきた。

| 年代 | 規格 | 速度 | 備考 |
|------|------|------|------|
| 1980 年代 | 各社独自（I-BUS / BEAM / MPCS など） | 〜10 kbps | メーカーごとにバラバラ |
| 1986〜 | **CAN** | 最大 1 Mbit/s | Bosch が提唱、業界標準へ集約 |
| 2003〜 | **LIN 2.0** | 最大 20 kbit/s | 低コスト補助バスとして普及 |
| 2004〜 | **FlexRay** | 最大 10 Mbit/s | X-by-Wire 向け。高コストで普及は限定的 |
| 2012〜 | **CAN FD** | 最大 8 Mbit/s | CAN の後継。ペイロード 64 バイト |
| 2015〜 | **100BASE-T1** | 100 Mbit/s | 車載 Ethernet 元年（IEEE 802.3bw） |
| 2016〜 | **1000BASE-T1** | 1 Gbit/s | ADAS・カメラ伝送に普及（IEEE 802.3bp） |
| 2018〜 | **CAN XL** | 最大 10 Mbit/s | CAN FD の後継（CiA 610、標準化進行中） |
| 2020〜 | **10GBASE-T1** | 10 Gbit/s | L4 自動運転・8K カメラ向け（IEEE 802.3ch） |
| 〜現在 | **Ethernet TSN** | — | 時刻同期・QoS を追加し決定論的遅延を実現 |

**特徴まとめ：**

| 項目 | 内容 |
|------|------|
| 最大通信速度 | 1 Mbit/s（CAN 2.0） |
| 最大ペイロード | 8 バイト |
| トポロジー | バス型（マルチマスター） |
| 差動信号 | CAN_H / CAN_L の 2 線 |
| 標準化 | ISO 11898-1（データリンク層）/ ISO 11898-2（物理層） |

<h2 id="s3">03. 物理層とバス構造</h2>

CAN は **差動信号（Differential Signaling）** を使う。
ノイズ耐性が高く、自動車のエンジンルームのような電気的ノイズの多い環境でも安定して動作する。

### 電圧レベル

| 状態 | CAN_H | CAN_L | 差動電圧 |
|------|-------|-------|----------|
| ドミナント（0） | ≈ 3.5 V | ≈ 1.5 V | ≈ 2.0 V |
| リセッシブ（1） | ≈ 2.5 V | ≈ 2.5 V | ≈ 0 V |

ドミナントはバスを「0」に強制する状態、リセッシブはバスを「1」にする（引っ張られていない）状態。
複数ノードが同時に送信した場合、ドミナントがリセッシブに勝つ。
この性質が後述の調停で重要になる。

### バス構造と終端抵抗

```
  Node A       Node B       Node C
    |             |             |
 [CANH]─────────────────────── [CANH]
 [CANL]─────────────────────── [CANL]
   │                             │
 [120Ω]                       [120Ω]   ← 終端抵抗（両端に必須）
```

バスの両端に **120 Ω の終端抵抗** を入れる。
これが欠けると信号の反射が起きて通信が不安定になるので注意。
ノード数が増えてもバスのインピーダンスを 60 Ω（並列合成）に保つのが目的。

<h2 id="s4">04. CAN フレームの種類と構造</h2>

CAN には 4 種類のフレームがある。

| フレーム | 用途 |
|----------|------|
| **Data Frame** | データを送信する通常フレーム |
| **Remote Frame** | 他ノードにデータ送信を要求する |
| **Error Frame** | エラー検出時に送信してバスをリセット |
| **Overload Frame** | 受信側が処理待ちを通知する（まれに使用） |

通常の通信は Data Frame がメイン。

### Data Frame の構造（標準フォーマット 11bit ID）

```
 SOF │   Arbitration  │ Control │    Data    │  CRC  │ ACK │ EOF │ IFS
  1  │ ID[10:0] + RTR │IDE r0 DLC│  0〜8 Byte│15+1bit│1+1  │  7  │  3  [bit]
```

| フィールド | ビット数 | 説明 |
|------------|---------|------|
| SOF | 1 | Start Of Frame。フレーム開始を示すドミナントビット |
| ID | 11（標準）/ 29（拡張） | メッセージの識別子。値が小さいほど高優先度 |
| RTR | 1 | Remote Transmission Request。Data=0, Remote=1 |
| IDE | 1 | ID Extended。標準=0, 拡張=1 |
| DLC | 4 | Data Length Code。データのバイト数（0〜8） |
| Data | 0〜64 bit | ペイロード（最大 8 バイト） |
| CRC | 15 + 1 | 誤り検出用 CRC とデリミタ |
| ACK | 1 + 1 | 受信成功ノードがドミナントで上書きする確認ビット |
| EOF | 7 | End Of Frame。すべてリセッシブ |
| IFS | 3 | Intermission。フレーム間スペース |

### ビットスタッフィング

CAN は NRZ（Non-Return-to-Zero）符号化を使う。
同じ値が 5 ビット以上連続すると同期が取れなくなるため、**5 ビット連続したら補数ビットを 1 つ挿入** するルールがある（Bit Stuffing）。
受信側は逆にデスタッフィングして元のデータを復元する。
CRC フィールドまでが Bit Stuffing の対象で、EOF 以降は対象外。

<h2 id="s5">05. 調停メカニズム（CSMA/CA）</h2>

CAN はマルチマスター方式なので、複数ノードが同時に送信を開始する可能性がある。
これを解決するのが **CSMA/CA（Carrier Sense Multiple Access / Collision Avoidance）**。

### 調停の仕組み

送信ノードは自分が送ったビットをバスから読み返す。
ドミナント（0）はリセッシブ（1）に勝つ特性を利用し、ID の上位ビットから 1 ビットずつ比較する。
自分が送った「1（リセッシブ）」がバス上では「0（ドミナント）」になっていたら、他のノードが勝っているとわかる。
負けたノードは即座に送信を中止し、バスが空くのを待ってリトライする。

```
Node A  ID: 0b011 0000 0001  →  送信優先度：高
Node B  ID: 0b011 0000 0101  →  送信優先度：低

bit 0〜7 は一致。bit 8 で A=0（ドミナント）、B=1（リセッシブ）
→ B はドミナントを検出して送信を中止。A が継続。
```

この方式は **非破壊的調停** と呼ばれる。
衝突しても A のフレームは破壊されず、そのまま送信が完了する。
イーサネットの CSMA/CD（衝突検出＆再送）と違い、再送ロスがない。

### 優先度の決まり方

**ID の数値が小さいほど優先度が高い。**
緊急性の高いメッセージには小さい ID を割り当てる設計が基本。

<h2 id="s6">06. エラー検出と Error State</h2>

CAN は通信の信頼性を高めるため、5 種類のエラー検出メカニズムを持つ。

### エラー検出の種類

| エラー種別 | 検出方法 |
|------------|---------|
| **Bit Error** | 送信ビットとバスの値が不一致（調停中を除く） |
| **Stuff Error** | 6 ビット以上同値が連続（ビットスタッフ違反） |
| **CRC Error** | 受信した CRC が計算値と不一致 |
| **Form Error** | EOF・デリミタなど固定フォーマット部が不正 |
| **ACK Error** | 送信したフレームに誰も ACK を返さなかった |

エラーを検出したノードは **Error Frame** を送信してバスに通知し、全ノードが現在のフレームを破棄してリカバリを行う。

### Error State（エラー状態の遷移）

各ノードは TEC（Transmit Error Counter）と REC（Receive Error Counter）を持つ。
エラー発生で増加し、正常送受信で減少する。
カウンタ値に応じて 3 つの状態を遷移する。

```
TEC / REC < 128        → Error Active   (正常。Active Error Frame を送信)
TEC or REC ≥ 128       → Error Passive  (Passive Error Frame を送信。バスへの影響が小さい)
TEC ≥ 256              → Bus Off        (バスから切り離し。送受信一切不可)
```

Bus Off から復帰するには、バス上で 128 × 11 ビットの連続リセッシブを検出する必要がある。
一部のノードが故障してもバス全体が巻き込まれにくい設計になっている。

<h2 id="s7">07. CAN FD とは</h2>

CAN FD（CAN with Flexible Data-rate）は、Bosch が 2012 年に策定した CAN の拡張規格。
ISO 11898-1:2015 に取り込まれ、現在は標準化されている。

### 登場背景

車載システムの高度化に伴い、ECU 間で扱うデータ量が爆増した。
ADAS（先進運転支援システム）やカメラ、OTA（Over-the-Air）アップデートなどは、CAN 2.0 の **8 バイト / 1 Mbit/s** では帯域が足りなくなってきた。
Ethernet に移行するほどでもないが、CAN の信頼性とコストは維持したい、という文脈で生まれたのが CAN FD。

### CAN 2.0 と CAN FD の比較

| 項目 | CAN 2.0 | CAN FD |
|------|---------|--------|
| 最大ペイロード | **8 バイト** | **64 バイト** |
| 調停フェーズ速度 | 最大 1 Mbit/s | 最大 1 Mbit/s（同じ） |
| データフェーズ速度 | 最大 1 Mbit/s | **最大 8 Mbit/s**（実装依存） |
| CRC 長 | 15 bit | 17 bit（≤16 byte）/ 21 bit（>16 byte） |
| ESI ビット | なし | あり（Error State Indicator） |

ポイントは **2 段階のビットレート**。
調停フェーズは従来と同じ低速で行い、調停が確定したデータフェーズだけ高速に切り替える。

<h2 id="s8">08. CAN FD フレーム構造と Bit Rate Switch</h2>

### CAN FD Data Frame の構造

```
 SOF │  Arbitration  │      Control        │    Data     │    CRC     │ ACK │ EOF
  1  │ ID + RRS(RTR) │IDE EDL(FDF) BRS ESI DLC│ 0〜64 Byte│17 or 21+1 │1+1  │  7  [bit]
                                ↑    ↑
                              FDF   BRS ← ここから高速フェーズ開始
```

| 追加フィールド | 説明 |
|---------------|------|
| **FDF（EDL）** | CAN FD Frame。このビットが 1 だと FD フレームとして扱う |
| **BRS** | Bit Rate Switch。1 のとき、このビット以降をデータフェーズ速度で送信 |
| **ESI** | Error State Indicator。送信ノードが Error Passive 状態なら 1 |

### ペイロード長と DLC の対応

CAN FD では DLC が 9〜15 の場合、バイト数が非線形にマッピングされる。

| DLC | バイト数 |
|-----|---------|
| 0〜8 | 0〜8（CAN と同じ） |
| 9 | 12 |
| 10 | 16 |
| 11 | 20 |
| 12 | 24 |
| 13 | 32 |
| 14 | 48 |
| 15 | 64 |

### Bit Rate Switch の動作

```
 ← Nominal Bitrate（例: 500 kbit/s）→  ← Data Bitrate（例: 2 Mbit/s）→  ← Nominal →

[SOF][Arbitration][Control...BRS] [ESI][Data][CRC] [ACK][EOF]
                            ↑高速開始                   ↑高速終了
```

BRS ビットのサンプルポイントで物理層トランシーバがビットレートを切り替える。
調停中はすべてのノードが同じ速度で張り付いているが、調停確定後は送信ノードだけが高速フェーズに入るイメージ。

<h2 id="s9">09. CAN FD の注意点</h2>

### 後方互換性

CAN FD フレームを **CAN 2.0 ノードは正しく受信できない。**
CAN 2.0 ノードは FDF ビットを Form Error と解釈し、Error Frame を送信してしまう。
CAN FD を導入する際はバス上のすべてのノードが FD 対応している必要がある。
（または CAN 2.0 ノードを FD フレームが流れるバスから分離する構成にする）

### クロック要件

データフェーズの高速化に伴い、**ノード間のクロック精度** がシビアになる。
CAN 2.0 では許容されていた水晶発振器のばらつきも、高速フェーズでは問題になりうる。
CAN FD 対応トランシーバ・コントローラを選定する際は、データシートのクロック精度要件を必ず確認すること。

### 伝送距離とビットレートのトレードオフ

| データビットレート | 最大バス長の目安 |
|-------------------|----------------|
| 1 Mbit/s | ≈ 40 m |
| 2 Mbit/s | ≈ 20 m |
| 5 Mbit/s | ≈ 10 m |
| 8 Mbit/s | ≈ 数 m |

高速になるほど伝播遅延の影響が大きくなり、バス長が制限される。
実際の設計では終端処理・ケーブル品質・スタブ長も含めて余裕を持たせること。

### CAN FD vs CAN XL

2018 年頃から CiA（CAN in Automation）/ Bosch が **CAN XL** の仕様策定を進めている。
ペイロード最大 2048 バイト、最大 10 Mbit/s を目指す規格だが、2026 年時点ではまだ普及途上。
CAN → CAN FD → CAN XL という進化のラインは覚えておいて損はない。

<h2 id="s10">10. 車載プロトコル比較</h2>

車載には CAN 以外にも複数の通信プロトコルがある。
用途・コスト・帯域のバランスで使い分けるのが実情。

| プロトコル | 速度 | トポロジー | ペイロード | 主な用途 | 特徴 |
|------------|------|-----------|-----------|---------|------|
| **LIN** | 最大 20 kbit/s | シングルマスター / スレーブ | 8 バイト | シート・ミラー・スイッチ類 | 安価・1 線・低速でよい箇所に |
| **CAN 2.0** | 最大 1 Mbit/s | バス型 | 8 バイト | パワートレイン・ボディ | 車載の基本。実績が厚い |
| **CAN FD** | 調停 1 Mbit/s・データ最大 8 Mbit/s | バス型 | 64 バイト | ADAS・OTA 対応 ECU | CAN の信頼性を維持しつつ高帯域化 |
| **FlexRay** | 最大 10 Mbit/s | バス / スター（デュアルチャネル） | 254 バイト | シャシー制御・X-by-Wire | 時分割・決定論的遅延が強み。高コスト |
| **Automotive Ethernet** | 100 Mbit/s〜10 Gbit/s | P2P / スイッチ | 大容量 | カメラ・OTA・中央演算 | 帯域最大。ハーネスが太い・EMC 対策が必要 |

### 使い分けの判断基準

```
                帯域が要る？
                    │
           No ──────┴────── Yes
           │                 │
      リアルタイム性？     決定論的？（FlexRay 領域）
           │                 │
     Yes        No      No ──┴── Yes
      │          │       │        │
    CAN / CAN FD  LIN  Ethernet  FlexRay
```

**実務での選定ポイント：**

- **LIN**: コスト最優先。
マスター ECU が CAN を喋り、スレーブ群には LIN でぶら下げるのが定番構成
- **CAN FD**: 既存 CAN インフラを活かしつつ帯域を上げたい場合の第一選択
- **FlexRay**: ステア・バイ・ワイヤ等の X-by-Wire で厳密な周期保証が必要な箇所。
ただし新規採用は減少傾向で CAN FD に置き換えられつつある
- **Automotive Ethernet（100BASE-T1 / 1000BASE-T1）**: 車体が 1 本の撚り線で Gigabit を通せる規格（IEEE 802.3bw / 802.3bp）。
OTA・ドライブレコーダー・高解像度カメラ伝送など大容量が必要な箇所

### Automotive Ethernet T1 シリーズの進化

T1 とは「1 本の撚り線（ツイストペア）で通信する」車載専用 Ethernet の総称。
通常の 2 線対ではなく 1 線対にすることでハーネス重量・コストを削減している。

| 規格 | 速度 | IEEE 規格 | 主な用途 |
|------|------|-----------|---------|
| 10BASE-T1S / T1L | 10 Mbit/s | 802.3cg | センサー・アクチュエータ向け低速バス |
| 100BASE-T1 | 100 Mbit/s | 802.3bw | ADAS 補助・ボディ系 ECU |
| 1000BASE-T1 | 1 Gbit/s | 802.3bp | カメラ・レーダー伝送 |
| 2.5GBASE-T1 | 2.5 Gbit/s | 802.3ch | 高解像度センサー統合 |
| 5GBASE-T1 | 5 Gbit/s | 802.3ch | 同上 |
| **10GBASE-T1** | **10 Gbit/s** | **802.3ch** | **L4 自動運転・8K カメラ・LiDAR 統合** |

IEEE 802.3ch は 2020 年承認。量産車への採用は 2022 年頃から始まりつつある。
将来的には中央集権型アーキテクチャ（Zone ECU / Central Computer）でのバックボーンとして使われる想定。

### Ethernet TSN（Time-Sensitive Networking）

通常の Ethernet はベストエフォート型で遅延を保証できない。
TSN は IEEE 802.1 ファミリーの拡張仕様群で、**時刻同期・帯域予約・優先制御** を追加して決定論的な遅延を実現する。

| TSN 主要規格 | 内容 |
|-------------|------|
| IEEE 802.1AS | 高精度時刻同期（gPTP） |
| IEEE 802.1Qbv | 時分割ゲーティング（TAS）。フレームを時間スロットで送信 |
| IEEE 802.1Qbu | フレームプリエンプション。優先フレームが割り込み可能 |
| IEEE 802.1CB | フレーム複製・除去による冗長化 |

CAN の強みだった「決定論的遅延」を Ethernet で実現しようというのが TSN の本質。
AUTOSAR Adaptive Platform や車載 Ethernet バックボーンへの統合が進んでいる。

### AUTOSAR との関係

AUTOSAR（AUTomotive Open System ARchitecture）は車載ソフトウェアのミドルウェア標準で、CAN / CAN FD / LIN / Ethernet をすべて **COM スタック** として抽象化する。
アプリ層は物理プロトコルを意識せず `Com_SendSignal()` / `Com_ReceiveSignal()` を呼ぶだけでよい設計になっている。
プロトコル変更の影響をアプリ層に波及させない、という点で実務上かなり助かる仕組み。

<h2 id="s11">11. その他の活用シーン</h2>

### 産業機器・ロボット

- **CANopen**（IEC 61158）: 産業用 CAN の上位プロトコル。
モーターコントローラ・PLC などで標準的に使われている。
- **DeviceNet**: 米国製造業向けの CAN ベースフィールドバス。
- ロボットアームの各軸 ECU 間通信など、リアルタイム性が求められる用途に向いている。

### 医療機器

患者監視システムや手術ロボットなど、高信頼性と決定論的な遅延が求められる分野でも採用されている。
高信頼性が求められる医療機器分野でも広く採用されている。

---

> **まとめ**
>
> - CAN は差動 2 線・マルチマスター・非破壊的調停の堅牢なプロトコル
> - ドミナント / リセッシブの性質が調停とエラー検出の両方を支えている
> - CAN FD は「調停は低速・データは高速」の 2 段階レートで帯域を拡張
> - 後方互換がないため、FD 導入はバス全体の対応状況を要確認
> - 車載では LIN / CAN / CAN FD / FlexRay / Ethernet を用途に応じて使い分ける
