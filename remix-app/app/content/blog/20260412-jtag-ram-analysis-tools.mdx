<h2 id="s1">01. JTAG とは</h2>

JTAG（Joint Test Action Group）は、**IEEE 1149.1** として標準化されたインターフェース規格。
1990 年に策定され、もともとは「プリント基板上の部品が正しく実装されているか検査する」ことを目的に作られた。

現在では転用が進み、**マイコン・SoC のデバッグインターフェース**として広く使われている。
ARM Cortex-M / Cortex-R / Cortex-A、RISC-V、MIPS など主要なプロセッサアーキテクチャのほぼすべてが JTAG をサポートしている。

### なぜ JTAG が必要か

組み込みソフトのデバッグが難しい理由は、**実行環境が閉じている**から。

- UART が使えないほど初期化前の問題（ブート直後のクラッシュなど）
- リアルタイム性が求められ、ログ出力すると動作が変わってしまう
- ハードウェアの問題なのかソフトウェアの問題なのか切り分けが必要

こういった状況で、**CPU を外から止めて・レジスタやメモリを読む**ことができるのが JTAG の強み。

<h2 id="s2">02. JTAG のアーキテクチャ</h2>

### 信号線

JTAG は最小 4 本の信号線で構成される。

| 信号名 | 方向 | 役割 |
|--------|------|------|
| **TCK** | ホスト → デバイス | クロック。この立ち上がり / 立ち下がりに同期してデータが動く |
| **TMS** | ホスト → デバイス | TAP ステートマシンの制御。1/0 の入力でステートを遷移させる |
| **TDI** | ホスト → デバイス | データ入力（シリアル） |
| **TDO** | デバイス → ホスト | データ出力（シリアル） |
| **TRST** | ホスト → デバイス | （オプション）TAP リセット |

```
ホスト（デバッガ）                 ターゲット（マイコン）
        │                                 │
   TCK ─┼────────────────────────────────→│
   TMS ─┼────────────────────────────────→│
   TDI ─┼────────────────────────────────→│
   TDO ─┼←───────────────────────────────│
        │                                 │
```

### TAP コントローラー

ターゲット内部には **TAP（Test Access Port）コントローラー** と呼ばれるステートマシンが存在する。
TMS の値を TCK に同期して入力することで、状態が遷移する。

主要なステートを抜粋すると：

```
         ┌────────────────────────────────────────┐
         │          Test-Logic-Reset              │ ← リセット状態（TMS=1 を 5 クロック）
         └────────────────┬───────────────────────┘
                  TMS=0   │
         ┌────────────────▼───────────────────────┐
         │               Run-Test/Idle            │ ← 通常の待機状態
         └────┬───────────────────────────────────┘
         TMS=1│
    ┌─────────▼─────────┐     ┌──────────────────┐
    │  Select-DR-Scan   │     │  Select-IR-Scan  │
    │  （データレジスタ）│     │ （命令レジスタ） │
    └─────────┬─────────┘     └──────────────────┘
              │
    ┌─────────▼─────────┐
    │   Capture-DR      │ ← DR にデータをキャプチャ
    │   Shift-DR        │ ← TDI→シフトレジスタ→TDO でデータ転送
    │   Update-DR       │ ← シフト結果を実際のレジスタに反映
    └───────────────────┘
```

- **IR（Instruction Register）**: 次に何をするかの命令を書き込む
- **DR（Data Register）**: 命令に応じたデータを読み書きする

JTAG 通信は常に「IR に命令 → DR にデータ」の繰り返しで成り立っている。

### デイジーチェーン接続

複数のデバイスを直列につなぐことができる（デイジーチェーン）。
前段の TDO が後段の TDI に接続され、ホストから見ると 1 本のシフトレジスタとして扱える。

```
ホスト → [デバイス A] → [デバイス B] → [デバイス C] → ホスト
  TDI      TDI  TDO       TDI  TDO       TDI  TDO      TDO
```

<h2 id="s3">03. バウンダリスキャン</h2>

JTAG の本来の用途は**基板実装検査**（バウンダリスキャン）。

各 IC の I/O ピンに「バウンダリスキャンセル」と呼ばれる小さなラッチ回路が挿入されており、JTAG 経由でこれらを直接操作することで、**CPU を動かさずにピンの状態を読み書き**できる。

```
                IC 内部ロジック
                      │
      ピン ─── [BSセル] ── 内部信号
                      │
                  TDI / TDO シフトレジスタ
```

これにより：
- はんだ不良（オープン / ショート）の検出
- 隣接するデバイス間の配線導通確認

といった検査が基板に電源を入れるだけで行える。
開発フェーズでは使う機会は少ないが、量産ラインや基板評価で活躍する。

<h2 id="s4">04. デバッガとしての JTAG</h2>

ARM アーキテクチャでは、JTAG 経由で **DAP（Debug Access Port）** にアクセスすることで CPU のデバッグ機能を利用できる。

```
デバッガ（PC）
    │
    │ JTAG / SWD
    ▼
  DAP（Debug Access Port）
    ├── AHB-AP（AHB Access Port） → メモリ・ペリフェラルへのアクセス
    └── JTAG-DP / SW-DP          → CoreSight デバッグ機能へのアクセス
         └── CPU コア内
              ├── DWT（Data Watchpoint and Trace）
              ├── ITM（Instrumentation Trace Macrocell）
              └── ETM（Embedded Trace Macrocell）
```

> **SWD（Serial Wire Debug）について**
> ARM Cortex-M では JTAG の代替として **SWD** という 2 線式プロトコルも使える。
> SWDIO（データ）+ SWDCLK（クロック）の 2 本だけで動作し、ピン数が少ない小型マイコンで多用される。
> J-Link・OpenOCD など主要ツールはどちらもサポートしている。

### halt / resume / step

デバッガが「一時停止」を指示すると、CPU は現在の命令を完了した後に **Halted 状態**に入る。
この状態でレジスタ・メモリを自由に読み書きできる。

```
デバッガ操作         CPU の状態
─────────────────────────────────────────
halt 指示      →   実行中 → Halted（停止）
read register  →   レジスタ値をそのまま取得
write memory   →   任意のアドレスに値を書き込み
step           →   1 命令だけ実行して再び Halted
resume         →   実行再開
```

### ブレークポイントの仕組み

| 種別 | 仕組み | 用途 |
|------|--------|------|
| **ハードウェアブレーク** | CPU 内の比較回路（BP ユニット）が PC と一致したら halt | Flash 上のコード、高頻度の箇所 |
| **ソフトウェアブレーク** | 対象アドレスの命令を `BKPT` 命令で上書き | RAM 上のコードのみ（Flash は書き換えコストが高い） |
| **ウォッチポイント** | DWT が特定アドレスへの読み書きを検出して halt | 変数が書き換わる瞬間を捉える |

ARM Cortex-M では通常 4〜8 個のハードウェアブレークポイントが使える。

<h2 id="s5">05. RAM 解析とは</h2>

CPU を halt した状態で、**RAM の内容をまとめて読み出す**のが RAM ダンプ（メモリダンプ）。

### 何がわかるか

| 取得できる情報 | 活用例 |
|---------------|--------|
| **スタックの内容** | クラッシュ直前の関数呼び出し履歴（コールスタック）を復元 |
| **変数の値** | 異常発生時のシステム状態を再現 |
| **ヒープの状態** | メモリリークやヒープ破壊の痕跡 |
| **コード（RAM 配置の場合）** | 動的に書き換えられたコードの確認 |

### ダンプのタイミング

| タイミング | 方法 |
|-----------|------|
| **任意のタイミング** | デバッガで halt して手動ダンプ |
| **クラッシュ直後** | HardFault ハンドラから RAM を UART 等で出力 |
| **特定条件成立時** | ウォッチポイントで自動 halt してダンプ |

車載 ECU 開発では、**RAM ダンプを NVM（不揮発メモリ）に保存**して後から解析する手法もある（フリーズフレームの拡張版のような使い方）。

### アドレスマップの把握が前提

ダンプの解析には**リンカマップファイル（.map）**が必要。
どの変数がどのアドレスに配置されているかがわかって初めて、16 進数の羅列が意味を持つ。

```
// リンカマップファイルの例（抜粋）
.bss            0x20000000  0x400
  g_systemState 0x20000000  0x20   system_ctrl.o
  g_errorCount  0x20000020  0x04   error_manager.o
```

<h2 id="s6">06. 主要ツール紹介</h2>

### ツール比較

| ツール | 種別 | 価格帯 | 特徴 |
|--------|------|--------|------|
| **OpenOCD** | OSS | 無料（ハードウェアアダプタは別途） | 対応アーキテクチャが広い・カスタマイズ性が高い |
| **SEGGER J-Link** | 商用 | アダプタ数万円〜・ソフト無料 | 安定性が高く普及率 No.1・J-Link GDB Server で GDB と連携 |
| **Lauterbach TRACE32** | 商用 | 数十〜数百万円 | 自動車業界標準・トレース機能が強力・AUTOSAR 対応 |
| **Renesas E1 / E2 / E2 Lite** | 商用 | E2 Lite は数千円〜・ソフト無料 | Renesas MCU 専用・e² studio / CS+ と統合 |
| **pyOCD** | OSS | 無料 | Python ベース・ARM Cortex-M 特化・スクリプト自動化に向く |

### OpenOCD

オープンソースの JTAG / SWD デバッグサーバー。
設定ファイル（.cfg）でターゲットとアダプタを指定し、GDB から接続して使う。

```bash
# OpenOCD 起動例（STM32F4 + ST-LINK の場合）
openocd \
  -f interface/stlink.cfg \
  -f target/stm32f4x.cfg

# 別ターミナルから GDB で接続
arm-none-eabi-gdb firmware.elf
(gdb) target remote :3333
(gdb) monitor halt
(gdb) info registers
```

コスト最優先の評価環境や OSS プロジェクトで重宝する。

### SEGGER J-Link

産業・車載を問わず最も普及しているデバッグプローブ。
専用の **J-Link GDB Server** を経由して GDB や IDE（IAR / Keil / VS Code + Cortex-Debug）と接続できる。

```bash
# J-Link GDB Server 起動例
JLinkGDBServer -device Cortex-M4 -if SWD -speed 4000

# RAM ダンプ（J-Link Commander から直接実行）
J-Link> halt
J-Link> savebin dump.bin 0x20000000 0x10000   # 64KB ダンプ
```

**J-Link RTT（Real-Time Transfer）** という機能も便利で、halt せずに printf 相当のログをリアルタイムで取得できる。
UART を使えない状況での軽量ログ手段として有用。

### Lauterbach TRACE32

自動車業界では事実上の標準デバッガ。
単なる halt / step デバッグにとどまらず、**CPU の命令トレースをリアルタイムで記録**できる。

```
TRACE32 の主な機能
├── JTAG / SWD / NEXUS / Aurora など多数対応
├── ETM / PTM トレース → プログラムフローの完全記録
├── AUTOSAR OS・RunTime対応（タスクスイッチの可視化）
├── FlashProgrammer → ECU へのフラッシュ書き込み
└── PRACTICE スクリプト → 解析自動化
```

TRACE32 では `.elf` ファイルを読み込んでシンボル情報を解決するのが標準的な使い方。
アドレスに対して変数名・関数名・ソースコード行が対応づけられ、RAM の内容を人間が読める形で表示できる。

```
; PRACTICE スクリプト例（起動〜シンボルロードまで）
SYStem.CPU CortexM4          ; ターゲット CPU を指定
SYStem.Up                    ; JTAG 接続・CPU リセット
Break.direct                 ; halt

; .elf を読み込んでシンボルテーブルを構築
; /NoCODE → コードの転送はしない（デバッグ情報だけ取得）
Data.LOAD.Elf firmware.elf /NoCODE

; シンボル名でメモリを表示
Var.View g_errorCount        ; 変数の値を表示
Var.View %address g_systemState ; アドレス付きで表示
```

`/NoCODE` オプションをつけないと `.elf` 内のバイナリをそのままターゲットに書き込もうとするため注意。
シンボル解決だけしたい場合は必ず `/NoCODE` を指定する。

価格は高いが、**「クラッシュ前の数秒間の実行ログを遡って見る」** ようなことができるのはトレース機能を持つツールだけ。
複雑な割り込み絡みのバグ追跡に強い。

### Renesas E1 / E2 / E2 Lite

Renesas マイコンを使う場合は、専用エミュレータの **E1 / E2 / E2 Lite** が標準の選択肢。
J-Link が ARM 向けの汎用ツールであるのに対し、E シリーズは Renesas アーキテクチャに最適化されている。

| エミュレータ | 位置づけ |
|-------------|---------|
| **E2 Lite** | 評価・個人開発向け。低コストで基本機能を提供 |
| **E2** | 量産開発向けの標準モデル。フラッシュ書き込み・デバッグ両対応 |
| **E1** | 旧世代。現在は E2 に置き換わりつつある |

Renesas MCU はデバッグインターフェースがアーキテクチャごとに異なる点が特徴。

| MCU ファミリ | デバッグ I/F | 信号線数 | 主な用途 |
|-------------|------------|---------|---------|
| **RH850**（車載向け） | JTAG / cJTAG | 4 本 / 2 本 | 車載 ECU のメイン CPU |
| **RX** | JTAG / FINE | 4 本 / 1 本 | 産業機器・家電 |
| **RL78** | TOOL0（FINE） | 1 本 | 小型マイコン・センサー系 |
| **RA**（ARM Cortex-M） | SWD / JTAG | 2〜4 本 | 比較的新しいシリーズ |

車載開発では **RH850** が多く、JTAG の他に **cJTAG（IEEE 1149.7）** という 2 線式の省ピン版も使われる。
cJTAG は TCK と TMS の 2 本だけで通信でき、ピンが少ない ECU コネクタ設計に有利。

統合開発環境は Renesas 純正の **e² studio**（Eclipse ベース）または **CS+** を使うのが一般的で、エミュレータのドライバも IDE に内包されているため、別途サーバーを立てる必要がない。

```
; e² studio でのデバッグ起動フロー（GUI 操作）
1. プロジェクト → デバッグ構成 → Renesas GDB Hardware Debugging
2. デバッグハードウェア: E2 を選択
3. ターゲット MCU: R7F701xxx（RH850）を指定
4. [デバッグ] ボタン → 自動で halt 状態で停止
```

J-Link は Renesas MCU にも対応しているが、**FINE / cJTAG は非対応のことが多い**。
Renesas 製品を触るなら E2 エミュレータを手元に置いておくのが無難。

<h2 id="s7">07. 実践フロー</h2>

### 基本的な RAM ダンプ取得フロー

```
1. ハードウェア接続
   └── JTAG / SWD ケーブルでプローブとターゲット基板を接続
       電圧レベルを確認（3.3V / 5V / 1.8V）

2. デバッグサーバー起動
   └── OpenOCD または J-Link GDB Server を起動

3. GDB 接続・halt
   $ arm-none-eabi-gdb firmware.elf
   (gdb) target remote :3333
   (gdb) monitor halt          # CPU を停止

4. レジスタ確認
   (gdb) info registers        # PC・SP・LR などを確認

5. RAM ダンプ取得
   (gdb) dump binary memory dump.bin 0x20000000 0x20010000

6. 解析
   └── バイナリエディタ / Python / addr2line でシンボル解決
```

### ダンプの解析例（Python）

```python
import struct

# .map ファイルから変数のアドレスを取得している前提
DUMP_BASE   = 0x20000000
VAR_ADDRESS = 0x20000020  # g_errorCount のアドレス
VAR_OFFSET  = VAR_ADDRESS - DUMP_BASE

with open("dump.bin", "rb") as f:
    data = f.read()

# uint32_t として読み出し（リトルエンディアン）
value = struct.unpack_from("<I", data, VAR_OFFSET)[0]
print(f"g_errorCount = {value}")
```

### クラッシュ解析の定石

HardFault が発生したとき、まず確認するのはスタック上の **例外フレーム**。
ARM Cortex-M では例外発生時に自動的に以下のレジスタがスタックに積まれる。

```
スタック（SP が指す位置から）
┌─────────┬────────────────────────────┐
│ +0x00   │ R0                         │
│ +0x04   │ R1                         │
│ +0x08   │ R2                         │
│ +0x0C   │ R3                         │
│ +0x10   │ R12                        │
│ +0x14   │ LR（リターンアドレス）      │
│ +0x18   │ PC（クラッシュした命令）    │ ← ここが最重要
│ +0x1C   │ xPSR                       │
└─────────┴────────────────────────────┘
```

ダンプから SP の値を読み、その位置から 0x18 バイト後のアドレスを `addr2line` に通せば、**何行目でクラッシュしたか**がわかる。

```bash
arm-none-eabi-addr2line -e firmware.elf 0x08012345
# → src/motor_ctrl.c:87
```

<h2 id="s8">08. よくある躓きポイント</h2>

### 電圧レベルの不一致

プローブの I/O 電圧とターゲットの電源電圧が合っていないと接続できない（最悪、基板を破損）。

| ターゲット電源 | プローブ設定 |
|---------------|-------------|
| 3.3V | 3.3V（多くのプローブで自動検出） |
| 1.8V | 1.8V に手動設定が必要なことが多い |
| 5V | 現代のマイコンでは稀だが古い製品に存在 |

J-Link は **Vtref ピン**でターゲット電圧を自動検出して追従するものが多いが、確認は必須。

### クロック（TCK/SWCLK）速度

高速すぎると通信エラーになる。
**基板のパターン長・容量によって上限が変わる**ため、まず低速（400kHz 程度）で接続を確認してから上げていく。

```bash
# OpenOCD でのクロック設定
adapter speed 400      # 400kHz から試す
```

### フラッシュ読み出し保護（RDP / Secure Boot）

MCU によってはフラッシュ読み出し保護が有効になっており、JTAG 経由のアクセスが制限・無効化されていることがある。
開発中は保護が無効な開発用ビルドを使うこと。
量産設定を誤って開発機に書き込むと復元できなくなるケースもあるため注意。

### フラッシュの書き換え寿命

JTAG デバッグ中に**フラッシュの寿命を知らず知らず消費している**ことがある。
NOR フラッシュの書き換え保証回数は一般的に **10,000〜100,000 回**程度で、超えると書き込み不良や読み出し化けが発生しうる。

寿命を縮める主な原因は**ソフトウェアブレークポイント**。
ソフトウェアブレークは対象アドレスの命令を `BKPT` 命令で上書きする仕組みのため、ブレークのセット・解除のたびにフラッシュの消去 → 書き込みが発生する。

```
ソフトウェアブレーク 1 回の操作
  1. 対象セクタを消去（フラッシュ消去 1 回）
  2. BKPT 命令を書き込み（フラッシュ書き込み 1 回）
  ↕ デバッグ中...
  3. 元の命令に戻すため再度消去 → 書き込み（さらに 1 サイクル消費）
```

開発中に何百回もブレークポイントを付け替えると、特定セクタへの書き込みが集中して局所的に寿命が縮む。

#### 対策

| 対策 | 内容 |
|------|------|
| **ハードウェアブレークを優先する** | CPU 内の比較回路で halt するためフラッシュに一切触れない。残数（4〜8 個）の範囲で積極的に使う |
| **RAM にコードを配置してデバッグ** | 集中的にデバッグしたい関数を RAM 上で動かせば、ソフトウェアブレークを使っても寿命に影響しない |
| **不要なブレークを残さない** | セッション終了時にブレークポイントをすべて解除してから切断する |
| **書き換えが頻繁な評価基板は使い捨て前提で管理** | 開発後期の評価基板と、デイリーのデバッグ用基板を分けておくと安心 |

フラッシュ寿命の問題は**症状が出るまで気づきにくい**。
特定の条件下でだけ動作が不安定になり、フラッシュ劣化が原因だと判明するまで時間がかかるケースもある。
ハードウェアブレークを基本とする習慣をつけておくのが無難。

### GDB が「No remote」と返す

デバッグサーバー（OpenOCD / J-Link GDB Server）が起動しているか、ポート番号が合っているかを確認。

```bash
# よくある原因
# 1. サーバーが起動前に GDB を起動した → サーバーを先に起動する
# 2. ファイアウォールがポートをブロック → ローカル接続なら基本問題ない
# 3. ターゲットへの電源が入っていない → 当然だが意外と多い
```

---

> **まとめ**
>
> - JTAG は IEEE 1149.1 の規格で、TAP コントローラーを通じて CPU のデバッグ機能にアクセスできる
> - ARM では DAP 経由で halt / step / ブレークポイント / メモリ読み書きが可能
> - RAM ダンプとリンカマップを組み合わせることで、クラッシュ時の変数状態やスタックを解析できる
> - ツールは用途と予算で選ぶ：OSS なら OpenOCD / pyOCD、安定性重視なら J-Link、トレース重視なら TRACE32
> - 接続時は電圧レベル・クロック速度・読み出し保護を必ず確認する
